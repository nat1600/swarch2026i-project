## 2. Guia técnica.

### Proxy reverso, definición y propósito:

Un proxy reverso es un componente que controla la entrada de requests a una red; gracias a esto se puede determinar a cuál servicio interno le corresponde cada request. Uno de los más utilizados es nginx, que además permite implementar el patrón "Secure Channel", el cual consiste en añadir TLS a la comunicación. Sin embargo, en este documento solo se mostrará su uso como proxy reverso.

Entre sus usos más comunes se encuentran: load balancing (distribuir requests entre varias instancias del mismo servicio), caching (almacenar elementos estáticos de la página web para servirlos más rápido), seguridad (terminación TLS mencionada anteriormente) y aceleración web (optimizar la entrega de paquetes hacia el cliente).

### Como implementarlo:

#### Paso 1 — Crear la estructura de carpetas para nginx

Se creó la carpeta `nginx/` en la raíz del proyecto. Dentro de ella se creó la carpeta `certs/` (donde se almacenan los certificados TLS) y el archivo `nginx.conf` (la configuración del proxy reverso).

```
nginx/
├── certs/
│   ├── cert.pem
│   └── key.pem
└── nginx.conf
```

#### Paso 2 — Generar el certificado TLS autofirmado

Desde la carpeta `nginx/certs/`, se ejecutó el siguiente comando para generar un certificado autofirmado válido para desarrollo local:

```bash
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout key.pem \
  -out cert.pem \
  -subj "/CN=localhost" \
  -addext "subjectAltName=DNS:localhost,IP:127.0.0.1"
```

Significado de cada parámetro:

| Parámetro                      | Descripción                                                                                                                                   |
|--------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------|
| `req -x509`                    | Genera directamente un certificado autofirmado (sin pasar por una CA externa)                                                                 |
| `-nodes`                       | No cifra la clave privada con contraseña (necesario para que nginx la lea sin intervención manual al arrancar)                                |
| `-days 365`                    | El certificado es válido por 365 días                                                                                                         |
| `-newkey rsa:2048`             | Crea una nueva clave privada RSA de 2048 bits junto con el certificado                                                                        |
| `-keyout key.pem`              | Guarda la clave privada en `key.pem`                                                                                                          |
| `-out cert.pem`                | Guarda el certificado público en `cert.pem`                                                                                                   |
| `-subj "/CN=localhost"`        | Establece el Common Name del certificado como `localhost`                                                                                     |
| `-addext "subjectAltName=..."` | Agrega los Subject Alternative Names requeridos por navegadores modernos para aceptar el certificado como válido en `localhost` y `127.0.0.1` |

#### Paso 3 — Agregar el certificado al navegador (Brave)

Para que el navegador acepte el certificado autofirmado sin mostrar advertencias de seguridad, se importó `cert.pem` manualmente en la lista de autoridades de confianza de Brave. Si bien no es 100% necesario para la página web, si lo fue para la extensión.

#### Paso 4 — Configurar el `docker-compose.yml`

Se agregó el servicio `nginx` al `docker-compose.yml`, montando la configuración y los certificados como volúmenes de solo lectura, y exponiendo los puertos 80 (HTTP) y 443 (HTTPS):

```yaml
nginx:
  image: nginx:1.27-alpine
  container_name: nginx
  ports:
    - "80:80"
    - "443:443"
  volumes:
    - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
    - ./nginx/certs:/etc/nginx/certs:ro
  depends_on:
    - web-app
    - api-gateway
```

- `ports`: expone el puerto 80 para redirigir HTTP → HTTPS, y el 443 para recibir tráfico HTTPS.
- `volumes`: monta el archivo de configuración y la carpeta de certificados dentro del contenedor como solo lectura (`:ro`).
- `depends_on`: garantiza que nginx arranque después de `web-app` y `api-gateway`.

#### Paso 5 — Configurar `nginx.conf`

Se configuró el archivo `nginx/nginx.conf` con dos bloques `server`: uno para HTTPS (puerto 443) que actúa como proxy reverso, y otro para HTTP (puerto 80) que redirige todo el tráfico hacia HTTPS.

```nginx
events {
    worker_connections 1024;
}

http {
    server {
        listen 443 ssl;
        server_name localhost;

        ssl_certificate     /etc/nginx/certs/cert.pem;
        ssl_certificate_key /etc/nginx/certs/key.pem;

        proxy_buffer_size           32k;
        proxy_buffers               8 32k;
        proxy_busy_buffers_size     64k;
        large_client_header_buffers 4 16k;

        # Rutas de Auth0 SDK → Next.js
        location = /api/auth/login    { proxy_pass http://web-app:3000; }
        location = /api/auth/logout   { proxy_pass http://web-app:3000; }
        location = /api/auth/callback { proxy_pass http://web-app:3000; }
        location = /api/auth/profile  { proxy_pass http://web-app:3000; }
        location = /api/auth/token    { proxy_pass http://web-app:3000; }

        # Resto de rutas /api/ → api-gateway
        location /api/ {
            proxy_pass http://api-gateway:8080;
            proxy_set_header Host              $host;
            proxy_set_header X-Real-IP         $remote_addr;
            proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;

            proxy_buffer_size        16k;
            proxy_buffers            8 16k;
            proxy_busy_buffers_size  32k;
        }

        # Todo lo demás → Next.js
        location / {
            proxy_pass http://web-app:3000;
            proxy_set_header Host              $host;
            proxy_set_header X-Real-IP         $remote_addr;
            proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;

            proxy_buffer_size        16k;
            proxy_buffers            8 16k;
            proxy_busy_buffers_size  32k;
        }
    }

    # Redirige HTTP → HTTPS
    server {
        listen 80;
        server_name localhost;
        return 301 https://$host$request_uri;
    }
}
```

Durante las pruebas se detectaron errores `502 Bad Gateway` causados por cabeceras y respuestas demasiado grandes para los buffers por defecto de nginx (los tokens JWT de Auth0 son especialmente voluminosos). Para solucionarlo se ajustaron los tamaños de buffer a nivel de servidor y por bloque `location`:

| Directiva                     | Valor                                | Descripción                                                                                                       |
|-------------------------------|--------------------------------------|-------------------------------------------------------------------------------------------------------------------|
| `proxy_buffer_size`           | `32k` (servidor) / `16k` (locations) | Tamaño del buffer para la primera parte de la respuesta del upstream (cabeceras)                                  |
| `proxy_buffers`               | `8 32k` / `8 16k`                    | Número y tamaño de los buffers para el cuerpo de la respuesta                                                     |
| `proxy_busy_buffers_size`     | `64k` / `32k`                        | Máximo de buffers que pueden estar enviando datos al cliente simultáneamente                                      |
| `large_client_header_buffers` | `4 16k`                              | Tamaño de los buffers para cabeceras de petición del cliente (necesario por el tamaño del JWT en cookies/headers) |

La lógica de enrutamiento funciona de la siguiente manera:

- **Rutas de Auth0** (`/api/auth/*`): se usan `location =` con coincidencia exacta para que las rutas del SDK de Auth0 lleguen directamente a Next.js, ya que ese flujo de autenticación es manejado internamente por la app web y no debe pasar por el gateway.
- **Rutas de API** (`/api/`): cualquier otra ruta bajo `/api/` se redirige al `api-gateway`, que se encarga de autenticar el JWT y hacer proxy hacia los microservicios internos. Se propagan las cabeceras `X-Real-IP`, `X-Forwarded-For` y `X-Forwarded-Proto` para que los servicios internos conozcan el origen real de la petición.
- **Resto** (`/`): cualquier otra ruta va a Next.js (`web-app`).
- **Puerto 80**: el segundo bloque `server` captura todo el tráfico HTTP sin cifrar y emite una redirección permanente (301) hacia la misma URL pero en HTTPS, garantizando que ningún cliente quede en texto plano.
